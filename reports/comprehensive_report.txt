PROBLEM STATEMENT
─────────────────
A large retail chain with 45 stores across three store tiers (A, B, C) struggles
with three core operational challenges:

  1. DEMAND UNCERTAINTY — Store managers cannot reliably predict upcoming sales,
     leading to overstocking (capital waste) or understocking (lost revenue).

  2. PROMOTION INEFFICIENCY — Markdown campaigns ($1–$5M annual spend) are
     deployed uniformly across all stores with no data-driven targeting.

  3. PERFORMANCE OPACITY — No clear framework exists to differentiate
     high-performing stores from underperformers or to tailor strategies.

REAL-WORLD USE CASES
────────────────────
  • Inventory Optimization   : Reduce overstock by 15% using 8-week forecasts
  • Demand Planning          : Auto-generate purchase orders when predicted
                               sales exceed safety stock threshold
  • Pricing Strategy         : Adjust prices dynamically based on CPI + local
                               unemployment signals
  • Promotion Planning       : Deploy markdowns only to price-sensitive segments
                               (Segment B & C stores) where lift > cost
  • Holiday Preparation      : Staff up 6 weeks ahead using forecast output

INVENTORY MANAGEMENT STRATEGY
──────────────────────────────
  1. FORECAST-DRIVEN REORDER
     • Use 8-week XGBoost forecast as primary demand signal
     • Set dynamic reorder points = forecast + safety_stock_buffer
     • Safety stock = 1.645 × demand_std × sqrt(lead_time_weeks)
     • Review and auto-trigger POs when forecast > current stock − reorder_pt

  2. HOLIDAY BUFFER PROTOCOL
     • 6 weeks before Thanksgiving: +25% stock for Depts 72, 90, 92, 95
     • 8 weeks before Christmas: +35% stock for top 20 departments by volume
     • Post-holiday: aggressive markdown on seasonal inventory within 2 weeks

  3. SEGMENT-BASED STOCK ALLOCATION
     • S-stores: Premium assortment depth; 20% broader SKU range
     • C-stores: Rationalize to top 70% of SKUs by velocity; eliminate tail

PRICING STRATEGY
─────────────────
  1. DYNAMIC PRICING TRIGGERS
     • CPI_Change > +0.3 in month → increase premium/private-label prices 1–2%
     • Unemployment_Change > +0.5 in month → activate value-tier promotions
     • Fuel_Price > $3.50 → promote delivery/online + increase margin on

PREPROCESSING QUESTIONS
────────────────────────
Q1.  Why fill MarkDown NaN with 0 instead of mean/median?
     ANS: NaN here is structurally missing (MNAR — Missing Not At Random). A
     missing markdown means no promotion occurred. The true value IS 0. Using
     mean would impute a fake promotional spend that never happened.

Q2.  What is the difference between MCAR, MAR, and MNAR?
     ANS: MCAR = missing completely at random (no pattern). MAR = missingness
     depends on observed data. MNAR = missingness depends on the missing value
     itself. MarkDowns are MNAR. CPI/Unemployment are MAR (missing at end of
     time series = MAR because it depends on date).

Q3.  How do you prevent data leakage in time-series feature engineering?
     ANS: All lag/rolling features must use .shift(1) or more — never use
     the current period's value. Train/test split must be strictly temporal
     (no random splits). Features derived from test data must never enter
     training (e.g., don't normalize using test set statistics).

Q4.  Why log-transform store size (LogSize) instead of using raw Size?
     ANS: Raw Size ranges from 34,000 to 219,000 sq ft — a 6× range. Tree
     models handle this fine, but distance-based models (K-Means) would be
     dominated by this large-scale feature. Log compression makes the scale
     comparable to other features and captures diminishing returns (a store
     twice as large is not twice as productive).

Q5.  What are lag features and why are they the most important predictors?
     ANS: Lag features use historical sales (e.g., 1 week ago, 4 weeks ago)
     as predictors of the current week. They capture temporal autocorrelation
     — the fact that this week's sales are strongly correlated with recent
     weeks. Sales_Lag1 alone explains ~31% of variance in this dataset.

ANOMALY DETECTION QUESTIONS
────────────────────────────
Q6.  What are the assumptions of Z-Score anomaly detection?
     ANS: Assumes data is approximately normally distributed. Computes
     (x − μ) / σ and flags if |z| > threshold (typically 3). Sensitive to
     existing outliers since they inflate σ, making the method less sensitive.
     Not suitable for highly skewed distributions without transformation.

Q7.  Why might IQR detect more anomalies than Z-Score?
     ANS: IQR (1.5×IQR rule) is more aggressive for skewed distributions
     like retail sales. Since sales are right-skewed (mean > median), the
     IQR whiskers extend to lower thresholds than z=3 on the right tail,
     flagging more "high sales" as anomalies.

Q8.  How does Isolation Forest work and why is it suited for retail data?
     ANS: It builds an ensemble of random decision trees. Anomalies are
     isolated faster (fewer splits needed) than normal points. It's
     multivariate — can detect unusual COMBINATIONS (e.g., high MarkDown
     + low sales, which individually might look normal). Requires no
     distributional assumptions. O(n log n) complexity — scales well.

Q9.  What is contamination parameter in Isolation Forest?
     ANS: Expected fraction of outliers in the dataset. Setting it to 0.02
     means we expect ~2% of data to be anomalous. This controls the decision
     threshold. If too low, we miss real anomalies; too high, excessive false
     positives. Should be set based on domain knowledge or calibrated against
     labeled data if available.

Q10. How do you decide which anomaly handling strategy to use?
     ANS: CAP: use when outliers are real but extreme (holiday spikes).
     REPLACE (median): use when outliers are likely errors. FLAG: use when
     you want to keep the data but inform downstream models. REMOVE: use
     only when outliers are confirmed data entry errors. For this project
     we cap because holiday anomalies are real and should be partially retained.

MACHINE LEARNING QUESTIONS
────────────────────────────
Q11. What is the difference between Random Forest and XGBoost?
     ANS: RF builds trees in parallel using bagging (bootstrap aggregation)
     and averages predictions. XGBoost builds trees sequentially, each
     correcting the errors of the previous (gradient boosting). XGBoost
     typically has lower bias; RF has lower variance. XGBoost is usually
     more accurate but requires more careful hyperparameter tuning.

Q12. Why use TimeSeriesSplit instead of random cross-validation?
     ANS: Random k-fold creates data leakage in time-series because future
     data ends up in the training fold, giving overly optimistic scores.
     TimeSeriesSplit always trains on past data and validates on future data,
     preserving temporal integrity. This gives a realistic estimate of how
     the model performs on unseen future weeks.

Q13. What does MAPE of 275% actually mean, and is it always bad?
     ANS: MAPE = mean(|actual - predicted| / actual) × 100. A 275% aggregate
     MAPE is inflated by departments with near-zero actual sales. If actual
     is $50 and prediction is $100, MAPE = 100%. For high-revenue departments
     (actual > $5,000), MAPE is typically 8–18%, which is very good for
     retail. Always evaluate MAPE conditionally — segment by sales volume.

Q14. How would you handle stores with no sales history (cold start)?
     ANS: (1) Use store type (A/B/C) as a proxy — assign the segment-average
     sales profile; (2) Transfer learning — start with the global model and
     fine-tune with new store's first 4–8 weeks of data; (3) Use store
     characteristics (size, type, region) to find the 3 most similar existing
     stores and average their sales patterns as a prior.

Q15. What hyperparameters matter most for XGBoost in retail forecasting?
     ANS: (1) n_estimators: more trees = better, but diminishing returns past
     500–1000; (2) learning_rate: lower is better (0.01–0.1), paired with more
     trees; (3) max_depth: 5–8 for tabular retail data; (4) subsample +
     colsample_bytree: 0.7–0.9 for regularization; (5) early_stopping_rounds:
     essential to prevent overfitting.

CLUSTERING QUESTIONS
────────────────────
Q16. How do you determine the optimal number of clusters?
     ANS: Use multiple methods: (1) Elbow Method — plot WCSS vs k, find the
     "elbow" where adding clusters gives diminishing reduction in WCSS;
     (2) Silhouette Score — measures cohesion vs separation, pick k with
     highest score; (3) Domain knowledge — 4 store tiers makes business sense;
     (4) Dendrogram for hierarchical clustering — cut at natural gaps.

Q17. What is the Silhouette Score formula and interpretation?
     ANS: s(i) = (b(i) − a(i)) / max(a(i), b(i)) where a(i) = avg distance
     to points in same cluster (cohesion) and b(i) = avg distance to nearest
     other cluster (separation). Range: −1 to +1. Score of 0.58 means
     clusters are reasonably well-separated with some overlap.

Q18. Why does K-Means fail with non-spherical clusters?
     ANS: K-Means minimizes Euclidean distance to centroids, implicitly
     assuming spherical, equally-sized clusters. For elongated, irregular,
     or varying-density clusters, DBSCAN or Gaussian Mixture Models (GMM)
     are more appropriate. Hierarchical clustering with Ward linkage handles
     non-spherical clusters better than K-Means.

Q19. What preprocessing is essential before K-Means?
     ANS: StandardScaler (or MinMaxScaler) is mandatory. K-Means uses
     Euclidean distance — a feature in range [0, 200,000] (store size) would
     dominate over a feature in range [0, 1] (MarkDown frequency). Scaling
     ensures all features contribute equally to the distance computation.

Q20. How would you validate that segments are meaningful for business?
     ANS: (1) Internal: Silhouette, Davies-Bouldin, Calinski-Harabasz;
     (2) External: Check if segments align with known labels (store types);
     (3) Business: Have domain experts review — do segments make intuitive
     sense? (4) Stability: Re-run with different random seeds; check if 80%+
     of assignments are stable. (5) Actionability: Can marketing/ops act
     differently on each segment?

TIME SERIES QUESTIONS
──────────────────────
Q21. What are the components of a time series?
     ANS: (1) Trend: long-term increase/decrease; (2) Seasonality: fixed
     periodic patterns (weekly, monthly, annual); (3) Cyclicality: irregular
     long-term fluctuations tied to business cycles; (4) Residual/Noise:
     random variation. Decomposition (STL/X-13) separates these for analysis.

Q22. What is the difference between ARIMA and SARIMA?
     ANS: ARIMA(p,d,q) handles trend and autocorrelation. SARIMA adds a
     seasonal component: (P,D,Q,s) where P=seasonal AR order, D=seasonal
     differencing, Q=seasonal MA order, s=period (52 for weekly). SARIMA is
     essential for retail data with strong annual seasonality.

Q23. How do you test for stationarity before fitting ARIMA?
     ANS: (1) Visual inspection of ACF/PACF plots; (2) ADF Test
     (Augmented Dickey-Fuller) — null hypothesis is unit root (non-stationary).
     p < 0.05 = stationary. (3) KPSS Test — null is stationary (complement).
     For retail sales: typically I(1) — first-order differencing (d=1) makes it
     stationary. Log transformation can stabilize variance first.

Q24. Why do we use weekly instead of daily granularity in this project?
     ANS: The raw data is weekly. Weekly aggregation: (1) aligns with retail
     replenishment cycles; (2) reduces noise from day-of-week effects;
     (3) computational efficiency; (4) most retail KPIs are reviewed weekly.
     Daily data would require additional day-of-week features and would have
     more missing/imputed values.

Q25. What is the "walk-forward" or "expanding window" validation approach?
     ANS: Instead of a single train/test split, we iteratively expand the
     training window and forecast the next period: Train on weeks 1–100,
     forecast week 101; then train on 1–101, forecast 102; etc. This gives
     a realistic estimate of rolling forecast accuracy and captures model
     degradation over time (concept drift). More computationally expensive
     but far more realistic than a single split.

ADVANCED QUESTIONS
───────────────────
Q31. How would you implement a hierarchical forecasting approach?
     ANS: Hierarchical forecasting respects the hierarchy: Total → Region →
     Store → Dept. Top-down: forecast total, then disaggregate using
     historical proportions. Bottom-up: forecast at lowest level, aggregate
     up. Middle-out: preferred method — forecast at store level, aggregate up
     (company) and disaggregate down (dept). Use reconciliation methods
     (MinT, OLS) to ensure coherent forecasts across levels.

Q32. What is concept drift and how do you detect it in retail?
     ANS: Concept drift = the statistical relationship between features and
     target changes over time (e.g., COVID changed consumer behavior). Types:
     Gradual drift (slow shift), Sudden drift (structural break), Seasonal
     drift (pattern shift). Detection: CUSUM test on prediction errors,
     Page-Hinkley test, DDM (Drift Detection Method). Mitigation: online
     learning, time-weighted training data, change point detection (ruptures).

Q33. Explain SHAP values and how you'd use them here.
     ANS: SHAP (SHapley Additive exPlanations) decomposes each prediction into
     individual feature contributions. For retail: "Store 23's prediction this
     week is $42K. It's $5K above baseline because: Sales_Lag1 (+$8K),
     IsHoliday (+$4K), offset by high Unemployment (-$3K), etc." Enables
     explaining forecasts to store managers in business language.

Q34. What is the difference between correlation and causation in this context?
     ANS: Unemployment is negatively correlated with sales (-0.12), but does
     not necessarily CAUSE lower sales — it's a confound (e.g., low-income
     areas have both higher unemployment AND smaller Type C stores). To
     establish causation, we'd need a controlled experiment — e.g., a store
     in a region where unemployment changed exogenously while a similar store's
     region did not. Correlation is sufficient for prediction; causation is
     needed for intervention design.

Q35. How would you scale this to a 10,000-store chain?
     ANS: (1) Distributed computing with Spark/Dask for preprocessing;
     (2) Global model with store-level embeddings (LightGBM or neural net)
     instead of separate models per store; (3) Feature store (Feast/Tecton)
     for shared feature computation; (4) Model serving with Kubernetes for
     parallel inference; (5) Approximate nearest neighbors for segmentation
     at scale (FAISS); (6) Streaming anomaly detection (Kafka + Flink).

Q36. What is the difference between precision and recall in anomaly detection?
     ANS: Precision = of all flagged anomalies, what % are truly anomalous?
     Recall = of all true anomalies, what % did we catch? In retail anomaly
     detection: high precision = few false alarms (ops teams prefer this);
     high recall = catch all real stockout signals (inventory teams prefer
     this). The consensus voting approach (2/3 methods agree) prioritizes
     precision — reducing false positive alerts to operations staff.

Q37. Why might a simpler model outperform XGBoost in certain conditions?
     ANS: XGBoost can overfit when: (1) Training data is small (<500 rows per
     entity); (2) Features are noisy or irrelevant; (3) Hyperparameters are
     poorly tuned. For Segment C stores with <$5K/week sales and sparse
     MarkDown data, a simple SARIMA or even linear model with seasonal dummies
     might outperform XGBoost because there's insufficient data to learn
     complex nonlinear patterns.

Q38. How do you handle class imbalance in anomaly detection?
     ANS: Anomalies are rare (~1–5%). Strategies: (1) Adjust contamination
     parameter in Isolation Forest to match expected prevalence; (2) Use
     anomaly scoring rather than hard labels — set threshold based on
     business tolerance; (3) SMOTE is NOT applicable (synthetic oversampling
     for supervised classification; anomaly detection is unsupervised);
     (4) Cost-sensitive evaluation — weight false negatives (missed anomalies)
     higher than false positives in the business context.

Q39. What does a high Sales_Lag1 importance mean practically?
     ANS: If last week's sales is the strongest predictor (importance 0.31),
     it means weekly sales have high autocorrelation — today's performance
     is strongly driven by recent performance. This implies: (1) momentum
     is real — a store doing well tends to continue; (2) short-term forecasts
     (1–2 weeks) are very reliable; (3) long-term forecasts (8+ weeks) will
     drift toward the mean as lag signal decays. This validates the use of
     rolling window features and motivates weekly model retraining.

Q40. How would you A/B test the effectiveness of segment-based marketing?
     ANS: (1) DEFINE: Segment B stores get targeted MarkDown2 campaigns;
     control = similar Segment B stores get standard blanket promotions;
     (2) RANDOMIZE: Assign stores randomly within Segment B to treatment/control;
     (3) DURATION: Run for 8–12 weeks to capture weekly seasonality;
     (4) MEASURE: Primary = Revenue lift per store/week; Secondary = Markdown
     ROI, Basket size; (5) ANALYZE: Difference-in-differences with store
     fixed effects to control for baseline differences; (6) DECISION:
     Roll out if p < 0.05 and incremental revenue > campaign cost.

COMMON MISTAKES IN RETAIL ANALYTICS
──────────────────────────────────────
  ✗ Using random train/test split on time-series data → data leakage
  ✗ Including future information in lag features → overly optimistic MAPE
  ✗ Evaluating MAPE on zero/near-zero sales rows → misleading 1000%+ MAPE
  ✗ Filling MarkDown NaN with mean → implies fake promotions happened
  ✗ Not scaling before K-Means → large features dominate clustering
  ✗ Choosing k based on elbow alone → always validate with silhouette
  ✗ Treating all anomalies as errors → holiday spikes are real and valid
  ✗ Using RMSE alone without MAE → RMSE penalizes large errors non-linearly
  ✗ Not considering seasonality in ARIMA → use SARIMA for retail
  ✗ One global model for all stores → store heterogeneity requires groups
